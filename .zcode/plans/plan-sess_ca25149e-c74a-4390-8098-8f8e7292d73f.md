# План remediation 4 skills (P0 + P1)

Применение meta-principles `writing-great-skills` (Predictability, Information Hierarchy, Invocation, Steering, Pruning) к 4 существующим skills. Без изменения функционального охвата — только чистка, дедупликация, добавление completion criterion и рефакторинг frontend-review.

## P0 — точечные правки (быстрый win)

### 1. `git-workflow-agent/SKILL.md` — удалить sediment §7.7 (ложный факт)
Реальный путь репо — `Arbibot-2` (дефис, без пробела), подтверждено `git rev-parse --show-toplevel`. Текущий текст врет: «Проект находится в `Arbibot 2` (путь с пробелом)».
- **§7.7**: удалить пункт про `nest start` truncation (устарел — репо уже на `tsc -p tsconfig.build.json`, см. AGENTS.md). Удалить/переписать пункт про пробел в пути: оставить только валидный совет «заключай пути с пробелами/дефисами в кавычки в `git diff`» как общее правило Windows, без ложного утверждения о текущем пути.

### 2. Дедупликация инвариантов `backend-review-agent` ↔ `architecture-guard-agent`
Проблема: блок `single-writer, reservation-first, versioned transitions, idempotent commit, outbox/inbox` повторяется дословно в обоих skills (failure mode: Duplication).
- **Решение без shared-файла**: skills зеркалируются в `~/.agents/skills/`, поэтому relative-path к `_shared/` ненадёжен. Вместо этого в `backend-review-agent/SKILL.md` сократить дословный дубль в разделе "Mandatory architecture constraints": заменить на указание, что архитектурные инварианты — primary authority `architecture-guard-agent`, а backend-review фокусируется на **implementation correctness** (state machines, event envelopes, DTO/schema, concurrency — то, чего нет в architecture-guard). Оставить 1-строчную сводку + ссылку на sibling skill (по имени, не relative path — так делают остальные skills в разделе "Integration with Other Skills").

## P1 — качественные улучшения

### 3. Добавить `## Completion criterion` во все 3 review-skills
`review-step.md` уже определяет criterion на уровне orchestrator (step 9: critical/major → `review_failed`; без блокеров → `review_passed` → `done`). Skills его не дублируют → риск Premature completion.
- Добавить в `architecture-guard-agent`, `backend-review-agent`, `frontend-review-agent` единый блок `## Completion criterion` со ссылкой на `review-step.md` (оркестратор) и конкретизацией для каждого skill: какие invariant-группы должны быть проверены, что считать sufficient для APPROVE (0 critical/architecture violations; все замечания с evidence). Verdict-связка: `APPROVE` ↔ `review_passed`; `REQUEST_CHANGES` ↔ `review_failed`.

### 4. Рефакторинг `frontend-review-agent/SKILL.md`
Самый слабый skill (failure modes: No-op generic rules, Negation-формулировки, Negative Space — query invalidation).
- **No-op → Arbibot-specific**: переписать generic React-правила ("строгую типизацию без any", "отсутствие fat page components") в affirmative Arbibot-specific checks, привязанные к реальным файлам (`apps/web/STACK-CONVENTIONS.md`, `lib/operator-query-keys.ts`).
- **Negation → Affirmative**: переформулировать ~40% правил из "отсутствие X" в "используй Y". Напр. "отсутствие дублирующих fetch patterns" → "все server state через BFF-маршруты (`/api/operator/*`) с `*_API_BASE` env".
- **Negative Space → заполнить**: добавить полноценную секцию `## Query invalidation checks` на основе `apps/web/QUERY_INVALIDATION.md` — конкретные ключи (`operatorKeys.paperTrades`, `settingsQueryKeys.configurations`), правила invalidation per-domain (dashboard/incidents/opportunities/execution/portfolio/paper/audit/settings), affirmative rules (гранулярная invalidation, `refetchOnWindowFocus: false`, optimistic pattern с rollback, Refresh button на read-only lists).
- **Убрать внутреннее противоречие**: "Не оценивай визуал «на вкус»" рядом с оценочными правилами — переформулировать в measurable checks.

### 5. `architecture-guard-agent` — удалить sediment
- Удалить/ослабить "Sequential execution: DEX-1 завершён до старта DEX-2" (проект feature-complete, DEX-2 done — правило устарело).
- "первая волна — Arbitrum, Base, BNB; DEX — Uniswap V2/V3, SushiSwap" — это планировочный артефакт, не architectural invariant. Перенести в замечание или удалить.
- Переписать Negation "использование any с ethers.js типами" → Affirmative "ethers.js v6 только с типами из `@arbibot/contracts-eth`".
