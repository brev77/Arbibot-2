# Outbox / Inbox pattern (P1-1.1-OIB)

## Outbox

- При изменении агрегата в **той же транзакции** вставляется строка в `outbox_events` с полным `envelope` и `payload`.
- Отдельный релей-процесс (worker) читает строки с `processed_at IS NULL`, публикует в Kafka/Redpanda, затем помечает `processed_at` (или использует at-least-once + идемпотентный consumer).
- Phase 1: релей может быть no-op; записи накапливаются для инспекции и будущего подключения брокера.

## Inbox

- Перед обработкой входящего сообщения consumer выполняет `INSERT` в `inbox_events (consumer_id, message_id)`; при конфликте уникального ключа — обработка пропускается (дубликат).
- Бизнес-эффект выполняется только после успешной «захватки» inbox.

## Идемпотентность

- `message_id` глобально уникален в outbox; повторная доставка не создаёт второй доменный эффект благодаря inbox на стороне получателя.
